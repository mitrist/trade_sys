# Протокол мультиагентной разработки (Dispatcher / Implement / QA)

Документ задаёт формат данных и шаблоны для автоматизации цикла разработки внутри Cursor. Один агент по очереди выполняет роли Dispatcher, Implement и QA.

---

## 1. Глобальный план (вход пользователя)

Пользователь в начале сессии передаёт план в одном из форматов.

### Вариант A: нумерованный список (рекомендуется)

```markdown
## Глобальный план

1. Создать файл config.py с настройками приложения.
2. Написать функцию подключения к БД в модуле db.py.
3. Написать тесты для функции подключения в test_db.py.
4. Добавить валидацию email в форму регистрации.
```

### Вариант B: структурированный JSON (для парсинга)

```json
{
  "plan": [
    "Создать файл config.py с настройками приложения",
    "Написать функцию подключения к БД в модуле db.py",
    "Написать тесты для функции подключения в test_db.py",
    "Добавить валидацию email в форму регистрации"
  ]
}
```

Агент (Dispatcher) извлекает из плана массив задач и выдаёт их по одной субагенту Implement.

---

## 2. Сообщение Implement → Dispatcher (готово к проверке)

После выполнения задачи Implement возвращает управление Dispatcher в структурированном виде:

```json
{
  "status": "ready",
  "task_index": 1,
  "task_text": "Написать функцию подключения к БД в модуле db.py",
  "changed_files": ["src/db.py"],
  "summary": "Добавлена функция get_connection() с обработкой ошибок и конфигом из config."
}
```

Или кратко в чате:

- «Задача выполнена, код готов к проверке. Изменённые файлы: `src/db.py`.»

Dispatcher передаёт в QA именно эти файлы/фрагменты для проверки по чек-листу.

---

## 3. Вердикт QA → Dispatcher (строго два варианта)

### ОК (все пункты чек-листа соблюдены)

```json
{
  "verdict": "OK"
}
```

### НЕ ОК (есть нарушение)

```json
{
  "verdict": "NOT_OK",
  "comment": "Нет обработки исключений при подключении к БД (п. 6 чек-листа)."
}
```

Поле `comment` обязательно при `NOT_OK`: кратко указать, какой пункт чек-листа нарушен (и при необходимости в чём именно).

---

## 4. Сообщение Dispatcher → Implement (при НЕ ОК)

Dispatcher формирует задачу на доработку:

```json
{
  "action": "fix",
  "task_index": 1,
  "task_text": "Написать функцию подключения к БД в модуле db.py",
  "qa_comment": "Нет обработки исключений при подключении к БД (п. 6 чек-листа).",
  "context_files": ["src/db.py"]
}
```

Или в свободной форме: «Исправить замечания QA: {qa_comment}. Файлы: {context_files}. Задача: {task_text}.»

Implement вносит правки в указанные файлы и снова сообщает «готово к проверке»; цикл QA → Dispatcher → Implement повторяется до вердикта «ОК».

---

## 5. Чек-лист QA (напоминание)

| № | Пункт | Критерий |
|---|--------|----------|
| 1 | Синтаксис и ошибки | Нет синтаксических ошибок и опечаток |
| 2 | Логика | Код соответствует задуманной логике задачи |
| 3 | Безопасность | Нет очевидных уязвимостей (SQL-инъекции, XSS и т.п.) |
| 4 | Производительность | Нет избыточных циклов и тяжёлых операций |
| 5 | Стиль кода | Соответствие стилю проекта и линтеру |
| 6 | Обработка ошибок | Есть базовая обработка исключений где нужно |

Вердикт «ОК» только если все шесть пунктов соблюдены.

---

## 6. Завершение работы

Когда все задачи из плана отмечены как выполненные (по вердикту «ОК»), Dispatcher сообщает пользователю, например:

«Все задачи глобального плана выполнены. Выполнено задач: 4. Изменённые файлы: config.py, db.py, test_db.py, forms.py.»

---

## 7. Как запустить процесс в Cursor

1. Открой чат (Chat) или Composer.
2. Убедись, что активна правило **multi-agent-dev-flow** (или скопируй сюда краткое описание процесса и чек-листа из правила).
3. Отправь сообщение в формате:

   **«Запусти мультиагентный цикл разработки по следующему плану:»**  
   [вставь глобальный план — нумерованный список или JSON.]

4. Агент поочерёдно будет: выдавать задачу Implement → проверять код как QA → при «НЕ ОК» возвращать задачу на доработку, при «ОК» переходить к следующей задаче до полного выполнения плана.
